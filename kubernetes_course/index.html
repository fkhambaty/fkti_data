ğŸ‘¶ What is Kubernetes? (Explain Like I'm 5)

Imagine you have a big toy box (your computer or the cloud). Inside, you want to run many small boats (these are called containers). Each boat runs one app: one runs a website, one runs a database, one runs your data pipeline.

Kubernetes is the captain that:

  • Starts the boats when you ask
  • Restarts a boat if it sinks (crashes)
  • Adds more boats when many people want to use your app
  • Gives each boat a name and address so others can find it

ğŸ”‘ Key words (simple)

Container = one small, runnable â€œboxâ€ (like a single boat).
Pod = the smallest unit Kubernetes runs; it can hold one or a few containers that always run together.
Cluster = the whole â€œseaâ€ where your boats run (your Mac with Docker Desktop, or a cloud).

Containers vs virtual machines

With VMs, you run several full â€œfake computersâ€ (each with its own OS) on one machine. With containers, you run many small, lightweight â€œboxesâ€ that share the host OS. Kubernetes schedules containers (wrapped in Pods), so you can pack more apps onto the same hardware and start/stop them in seconds.

VMs:  [ Host ] â†’ [ VM1: full OS + app ] [ VM2: full OS + app ] [ VM3: full OS + app ]
       One machine, 3 complete operating systems.

Containers: [ Host OS ] â†’ [ container A ] [ container B ] [ container C ] ...
            One OS, many isolated processes (containers). Kubernetes runs these.

ğŸ”„ Deployments â€“ Keep Your App Running and Scale It

ğŸ‘¶ In one sentence

A Deployment tells Kubernetes: â€œRun this many copies of this app (Pods), and if one dies, start another. If I change the image, roll out the new version.â€

Layman problem: â€œI want three copies of my API running. If one crashes, start a new one. When I push a new image, replace the old Pods gradually.â€ Thatâ€™s what a Deployment does. You set replicas: 3 and a Pod template; Kubernetes keeps 3 Pods running and replaces any that fail.

Example: run 3 copies of nginx.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: my-app
        image: nginx:latest

After kubectl apply -f deployment.yaml, run kubectl get pods. You'll see three Pods with names like my-app-7d8b9c5f6-xxxxx, all in Running state. Scale to 5 with: kubectl scale deployment my-app --replicas=5.

ğŸŒ Services â€“ A Stable Address for Your App

ğŸ‘¶ In one sentence

Pods get new IPs when they restart. A Service gives your app one stable name and IP. Other Pods (or you) call the Service; Kubernetes sends traffic to a healthy Pod behind it.

Problem: Your Deployment has 3 Pods, each with a different IP. You want â€œone addressâ€ (e.g. my-app) that always points to a healthy Pod. Create a Service that selects Pods by label (e.g. app: my-app). Other apps call http://my-app (or the Serviceâ€™s cluster IP); Kubernetes load-balances to a Pod.

Types: ClusterIP (default, internal only), NodePort (expose a port on each node), LoadBalancer (cloud load balancer). Example ClusterIP Service:

apiVersion: v1
kind: Service
metadata:
  name: my-app
spec:
  selector:
    app: my-app
  ports:
  - port: 80
    targetPort: 80

After applying, other Pods in the cluster can reach this app at http://my-app or http://my-app.default.svc.cluster.local.

ğŸ” ConfigMaps and Secrets

ConfigMaps store non-secret config (URLs, feature flags). Secrets store sensitive data (passwords, API keys). You mount them into Pods as files or env vars. Never put real secrets in your YAML or image!

Layman example: Your app needs a â€œdatabase URLâ€ and an â€œAPI key.â€ Put the URL in a ConfigMap and the API key in a Secret. In the Pod spec, inject them as environment variables or as files under /etc/config. If you change the ConfigMap/Secret, you can restart the Pod to pick up new values (or use a tool that does that for you).

Create a ConfigMap from a key-value pair:

In your Pod spec, use envFrom or volumeMounts to expose the ConfigMap to the container. For secrets, use kubectl create secret generic my-secret --from-literal=API_KEY=xxx and reference it the same way with secretRef or a secret volume.
