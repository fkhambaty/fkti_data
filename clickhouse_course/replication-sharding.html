Replication & Sharding

Scale ClickHouse from a single server to a distributed cluster. Understand replication for reliability and sharding for performance.

🔄 Replication

🧒 ELI5: The Backup Notebook

Imagine you're taking notes in class. Replication means your friend copies everything you write into their notebook too. If you lose your notebook, your friend still has all the notes. That's replication — keeping identical copies of data on multiple servers for safety.

How It Works

ClickHouse uses ClickHouse Keeper (or ZooKeeper) to coordinate replicas:

-- ReplicatedMergeTree = MergeTree + Replication
CREATE TABLE events_replicated (
    event_date Date,
    event_time DateTime,
    user_id UInt64,
    event_type String
) ENGINE = ReplicatedMergeTree(
    '/clickhouse/tables/{shard}/events',
    '{replica}'
)
PARTITION BY toYYYYMM(event_date)
ORDER BY (event_date, user_id);

⚠️ Replication ≠ Backup

Replication protects against server failure. But if you accidentally run DROP TABLE, it drops on ALL replicas! You still need proper backups.

🔀 Sharding

🧒 ELI5: The Pizza Slices

You have a pizza so big one person can't eat it. So you shard (slice) it — give 1/4 to each of 4 friends. Each friend eats their slice in parallel. Done 4x faster! Sharding splits your data across multiple servers so each one handles a fraction of the work.

-- Distributed table (queries all shards)
CREATE TABLE events_distributed
ENGINE = Distributed(
    'my_cluster',     -- cluster name
    'default',         -- database
    'events_local',    -- local table name
    rand()             -- sharding key
);
