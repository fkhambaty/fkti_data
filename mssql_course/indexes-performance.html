Indexes & Performance

Speed up queries with indexes; understand clustered vs nonclustered and execution plans.

What is an index?

An index is a structure that helps SQL Server find rows faster (like a book index). It can speed up WHERE, JOIN, and ORDER BY on the indexed columns. Indexes use storage and can slow down INSERT/UPDATE/DELETE, so create them on columns you search or sort by often.

Clustered vs nonclustered

TypeDescription
ClusteredOne per table. Stores the actual row data in index order. The table is physically ordered by the clustered index key (often the primary key).
NonclusteredSeparate structure that points to the data. You can have many nonclustered indexes per table.

CREATE INDEX & execution plans

Use CREATE INDEX to add a nonclustered index: CREATE INDEX IX_Orders_CustomerId ON Orders(CustomerId). Use execution plans (e.g. "Include Actual Execution Plan" in SSMS) to see how SQL Server runs the query and whether indexes are used.

Key Takeaway

Index columns used in WHERE, JOIN, and ORDER BY. One clustered index per table; add nonclustered indexes where they help. Check execution plans to verify.

How indexes help

Without an index: full table scan. With an index: seek or scan on the index, then lookup rows. Execution plan shows Index Seek (good) vs Table Scan (often costly on large tables).

CREATE INDEX example

-- Nonclustered index on CustomerId (speeds up WHERE CustomerId = ... and JOINs)
CREATE NONCLUSTERED INDEX IX_Orders_CustomerId ON Orders(CustomerId);

-- Composite index for filtering + ordering
CREATE NONCLUSTERED INDEX IX_Orders_CustomerId_OrderDate ON Orders(CustomerId, OrderDate DESC);

Practice

Create a nonclustered index on a column you often filter or join on. Run a query before and after; use "Include Actual Execution Plan" and compare (look for Index Seek vs Table Scan).
