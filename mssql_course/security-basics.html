Security Basics

Logins vs users, CREATE LOGIN/USER, GRANT REVOKE DENY, and roles.

Logins vs users

A login is a server-level identity (someone who can connect to SQL Server). A user is a database-level identity—linked to a login—that has permissions inside a specific database. One login can map to one user per database.

CREATE LOGIN & CREATE USER

CREATE LOGIN creates a server login (e.g. SQL or Windows auth). CREATE USER creates a database user and associates it with a login so that user can access that database.

GRANT, REVOKE, DENY & roles

GRANT gives a permission (e.g. SELECT on a table). REVOKE removes a granted permission. DENY explicitly blocks access (overrides GRANT). Roles group permissions; add users to roles (e.g. db_datareader) to give them the role's permissions.

Key Takeaway

Login = server access. User = database access. GRANT/REVOKE/DENY control what users can do. Use roles to manage permissions in bulk.

Security layers

Server (login) → Database (user) → Securables (tables, views). Permissions (GRANT/REVOKE/DENY) apply to users or roles on securables.

CREATE LOGIN and GRANT example

-- Create a SQL login
CREATE LOGIN AppUser WITH PASSWORD = 'StrongP@ss123';

-- In the target database: create user linked to login
USE MyDB;
CREATE USER AppUser FOR LOGIN AppUser;

-- Grant SELECT on a table
GRANT SELECT ON dbo.Orders TO AppUser;

-- Revoke later if needed
REVOKE SELECT ON dbo.Orders TO AppUser;

Practice

On a dev database: CREATE LOGIN and CREATE USER, then GRANT SELECT on one table. Connect with that login and run SELECT; try UPDATE and confirm it's denied unless you grant it.
